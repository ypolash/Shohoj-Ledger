# Version 1.2 AI Intelligence & Analytics Foundation

## Architecture

This phase introduces a robust data structure required for true AI-driven business intelligence, predictions, and operational insights without bogging down transactional tables. It operates independently as an overlay on top of existing modules (Finance, Inventory, HR, etc.).

## Models Created

### 1. `AnalyticsSnapshot`
- **Purpose**: A time-series caching layer. Instead of running expensive `SUM(amount)` over millions of ledger entries in real-time, scheduled jobs (from Workflow Engine Phase 2) will aggregate data overnight and store them here as point-in-time metrics.
- **Fields**: `id`, `companyId`, `metricKey`, `metricValue`, `metadata`, `periodStart`, `periodEnd`, `createdAt`.

### 2. `BusinessInsight`
- **Purpose**: Stores actionable observations. An insight might be generated by an anomaly detection rule ("Expense in category X spiked by 400%") or an AI summary.
- **Fields**: `id`, `companyId`, `type`, `title`, `description`, `severity`, `metadata`, `isRead`, `userId`, `createdAt`.

### 3. `ForecastRecord`
- **Purpose**: Stores the result of predictive ML models (e.g., "Predicted Revenue for Next Month = $45,000 with 85% confidence").
- **Fields**: `id`, `companyId`, `metricKey`, `forecastPeriod`, `predictedValue`, `confidence`, `metadata`, `createdAt`.

## Future AI Integration

Currently, the tables only store the output. In future phases, these tables will be populated by:
1. **Cron Data Pipelines**: Fetching raw data, grouping it, and saving to `AnalyticsSnapshot`.
2. **AI Inference Pipelines**: Pushing data chunks to an LLM or specialized ML model, calculating anomaly confidence scores, and writing to `BusinessInsight` or `ForecastRecord`.
3. **Chatbot Interface**: A Natural Language engine will query `AnalyticsSnapshot` to answer user questions like "How much did we spend on payroll last month?".

## Migration Strategy

No historical migration of data is required for these tables because they represent derived, analytical data. When the pipeline features are activated in a future phase, a one-off "Backfill Job" will simply calculate snapshots for all previous months based on the existing transactional logs (`LedgerEntry`, `Payroll`, `Attendance`).

## Risks

- **Data Staleness**: If the background snapshot worker fails, the AI will produce insights based on outdated numbers. (Mitigation: Add freshness checks via `createdAt` and use the `NotificationQueue` to alert the company admin if analytics fall behind).
- **Table Bloat**: Daily snapshots for hundreds of metric keys across thousands of companies could cause significant database bloat. (Mitigation: Aggressive partitioning, pruning old snapshots by rolling them up into monthly/yearly averages, and utilizing the `metadata Json` field to pack multiple related metrics).
