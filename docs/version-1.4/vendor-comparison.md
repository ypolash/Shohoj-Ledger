# Enterprise Vendor Comparison & Approval (Version 1.4 Phase 4)

## Architecture Overview
The Vendor Comparison engine acts as the analytical bridge between Vendor Quotations and final Purchase Order generation. It mathematically compares multiple supplier bids against a single Request For Quotation (RFQ).

## Data Models
1. **`VendorComparison`**: The header document capturing the overall decision matrix for an RFQ, including the chosen `evaluationMethod`, final `selectedSupplierId`, and approval tracking.
2. **`VendorComparisonItem`**: Represents a single supplier's bid metrics (total price, lead time, payment terms) extracted from their `VendorQuotation`. 

## Evaluation Methods
The system is built to support dynamic evaluation criteria via the `EvaluationMethod` enum:
- **`LOWEST_PRICE`**: Algorithmically ranks items strictly by `totalPrice`.
- **`WEIGHTED_SCORE`**: Evaluates based on a composite of `qualityScore`, `deliveryScore`, and `priceScore` to calculate an `overallScore`.
- **`BEST_VALUE`**: An advanced matrix balancing price and non-price terms (like payment terms).
- **`MANUAL_SELECTION`**: Allows procurement managers to bypass algorithmic ranking (useful for strategic sourcing or emergency buys).

## Workflow & State Machine
1. **DRAFT**: Comparison matrix is generated by pulling `SUBMITTED` quotes from an RFQ.
2. **UNDER_REVIEW**: The matrix is actively being scored or audited by a team.
3. **APPROVED**: Managerial sign-off is granted to the `selectedSupplierId`. This is a mandatory step before PO generation.
4. **REJECTED**: The comparison is deemed invalid or requires re-bidding.
5. **CONVERTED**: The selected quotation has been successfully transformed into a binding Purchase Order.

## Security & RBAC
- **Strict Tenancy:** The engine deeply enforces `{ companyId }` verification.
- **RFQ Lock:** Vendor comparisons cannot be initiated unless the underlying RFQ is strictly in the `CLOSED` state (meaning the bidding window has officially ended, ensuring fairness).
- **RBAC Scaffold:** Requires `VENDORCOMPARISON_VIEW`, `VENDORCOMPARISON_CREATE`, `VENDORCOMPARISON_APPROVE`, and `VENDORCOMPARISON_CONVERT`.

## Future Extensibility (AI Integration)
The `calculateScores()` methodology is designed as a pluggable service. In future phases, this can be extended to call a Machine Learning model (or LLM) that reads the `remarks` of each quotation, compares historical delivery SLA performance from the Supplier Master, and automatically populates the `overallScore`.
